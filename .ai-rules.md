# .ai-rules.md

本檔定義 AI agent 在此專案中的硬性約束。
若與 README、steering 或其他文件衝突，以本檔為準。

違反這些規則，可能造成：

- Amplify schema 或 function 接線失效
- 訂單狀態與明細狀態不同步
- 庫存扣減或回補錯誤
- 部署後資料不相容

## 必讀順序

開始工作前，至少依序閱讀：

1. 本檔 `.ai-rules.md`
2. `agents.md`
3. `.kiro/steering/product.md`
4. `.kiro/steering/structure.md`
5. `.kiro/steering/tech.md`

依任務類型再補讀：

- 前端：`.kiro/steering/frontend.md`
- 後端 / Amplify：`.kiro/steering/backend.md`

若任務涉及高風險流程，還要直接閱讀實際程式碼：

- `amplify/data/resource.ts`
- `amplify/backend.ts`
- 對應的 `amplify/functions/*/handler.ts` 與 `resource.ts`
- 對應的 `shared/logic/*` 與 `shared/models/*`

## 不可違反的約束

| 約束 | 說明 |
| --- | --- |
| 不做破壞性 schema 變更 | 不改既有 model 名稱、identifier、欄位型別、secondary index 名稱、`queryField` 名稱，除非明確說明資料影響與部署策略 |
| 訂單主狀態以共享邏輯為準 | `Order.status` 的推導與驗證必須依賴 `shared/logic/order-status` |
| 訂單明細狀態以共享邏輯為準 | `OrderItem.status` 的轉換必須依賴 `shared/logic/order-item-status` |
| 庫存操作必須先驗證 | 出貨、取消出貨、入庫、取消入庫等流程必須先驗證數量與狀態 |
| 合併 / 分拆訂單必須數量守恆 | 不可平白新增或遺失訂單明細數量 |
| 多筆同步更新必須維持原子性 | 涉及訂單、明細、商品、庫存的同步更新，不可只成功一部分 |
| Lambda 接線必須成套 | handler 改了，要一併檢查 `resource.ts`、`amplify/data/resource.ts`、`amplify/backend.ts`、`amplify/storage/resource.ts` |
| `AuthProvider` 是唯一認證來源 | 不建立第二套 auth context 或登入狀態 |
| UI 文案與錯誤訊息使用繁體中文 | 使用者看得到的內容都必須是繁體中文 |

## 高風險區域

| 觸及範圍 | 必讀 |
| --- | --- |
| schema / model / index / queryField / custom mutation | `amplify/data/resource.ts` |
| 訂單狀態、明細狀態、出貨 / 入庫 / 缺貨 / 取消流程 | `shared/logic/order-status.ts`、`shared/logic/order-item-status.ts`、`shared/logic/shipment.ts`、對應 `amplify/functions/*/handler.ts` |
| 訂單合併 / 分拆 | `amplify/functions/merge-orders/handler.ts`、`amplify/functions/split-order/handler.ts` |
| DynamoDB 交易 / 表格環境變數 / IAM | `amplify/backend.ts` |
| S3 商品照片 / 縮圖 | `amplify/storage/resource.ts`、`amplify/functions/generate-thumbnail/*` |
| 認證與受保護頁面 | `src/auth/AuthProvider.tsx`、`src/lib/route-guards.ts`、對應 route 的 `beforeLoad` |

## 禁止事項

- 不要手動編輯 `src/routeTree.gen.ts`
- 不要手動編輯 `amplify_outputs.json`
- 不要改既有 model 的 identifier 或 primary key 結構
- 不要刪除既有欄位，或任意變更欄位型別與 enum 語意
- 不要改既有 secondary index 名稱或 `queryField` 名稱
- 不要在 component 中重寫訂單狀態、庫存計算或其他核心商業規則
- 不要只改前端顯示，卻不檢查後端 mutation、shared logic 與資料來源
- 不要只改 Lambda handler，卻漏掉環境變數、grant、IAM 或 storage access
- 不要繞過 `shared/logic/` 在 Lambda 內平行重寫狀態機
- 不要新增 Redux、Zustand 或其他全域狀態管理庫
- 不要使用 `styled()` 或外部 CSS 檔案取代既有 MUI `sx` 慣例
- 不要建立第二套 auth 流程或額外 auth provider

## Schema 變更前必答

1. 會不會讓既有資料失效或無法讀取？
2. 會不會造成 table、index 或 `queryField` 重建 / 改名？
3. 會不會需要 backfill 或 migration？
4. 能不能用新增欄位、補新 index、補新 mutation 取代直接修改舊結構？

若無法清楚回答，就不要直接改 schema。

## 訂單與庫存流程檢查清單

修改以下任一流程時，必須一起檢查資料一致性：

- `confirm-purchase`
- `cancel-purchase`
- `confirm-received`
- `cancel-received`
- `confirm-shipment`
- `cancel-shipment`
- `confirm-out-of-stock`
- `cancel-out-of-stock`
- `merge-orders`
- `split-order`

額外注意：

- 任何同時更新 `Order`、`OrderItem`、`Product`、`ProductVariant` 的流程，都要維持原子性
- 訂單狀態變更必須驗證目前狀態是否合法，不可直接覆蓋
- 庫存不可扣成負數；取消流程要能正確回補
- 若流程依賴查詢 `OrderItem.byOrderId`，要確認 index 權限沒有漏授

## Amplify / Lambda 修改規則

- 新增 function 時，必須同步建立 `handler.ts` 與 `resource.ts`
- 新增後端資源後，必須在 `amplify/backend.ts` 註冊
- 若 handler 直接操作 DynamoDB、S3 或其他 AWS 資源，必須同步補齊：
  - `amplify/backend.ts` 的環境變數與 IAM / grant
  - `amplify/data/resource.ts` 的 handler 綁定與授權
  - `amplify/storage/resource.ts` 的 storage access
- 若使用 `$amplify/env/<name>`，名稱必須與對應 function 資源一致
- 若 handler 不需要 Amplify data client，不要硬加 `$amplify/env` 或 `getAmplifyDataClientConfig`
- 既有訂單流程 Lambda 多數以結構化 JSON 字串回傳結果；修改同一族 handler 時，應維持既有回傳契約

---

## 前端資料存取規則

- 受保護路由必須在 `beforeLoad` 檢查 `context.auth.isAuthenticated`
- 伺服器狀態統一透過 TanStack Query 管理，不要在 component 中散落查詢與 mutation 邏輯
- 型別優先從 schema 或 `shared/models` 衍生，不要手寫平行型別
- 錯誤處理應從 `catch` 提取 `Error.message`，並以繁體中文顯示於 MUI `Alert`
- 與訂單、商品、客戶、供應商相關的跨畫面規則，應抽到 `shared/logic/`

## 常見陷阱

| 症狀 | 常見原因 |
| --- | --- |
| `$amplify/env/<name>` 無法解析 | function 名稱與 `defineFunction({ name })` 不一致，或資源未正確註冊 |
| build 時找不到型別或 handler 綁定錯誤 | 新 function 沒有在 `amplify/backend.ts` 或 `amplify/data/resource.ts` 正確接線 |
| Lambda 可以讀表但 Query GSI 失敗 | `grantReadWriteData` 不包含 index ARN，需要額外補 `dynamodb:Query` 到 `.../index/*` |
| 訂單狀態與明細狀態不同步 | 跳過 shared logic，直接寫資料 |
| 庫存出現負數或回補錯誤 | 忽略數量驗證、取消流程或交易原子性 |
| 前端畫面正常但部署後 mutation 壞掉 | 只改了 UI 或型別，沒有同步更新後端 schema / function / authorization |
