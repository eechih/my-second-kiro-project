# 需求文件

## 簡介

本文件定義電子商務訂單、進貨（採購入庫）、出貨（銷售出庫）管理系統的需求。系統建構於現有 React 19 + AWS Amplify Gen2 專案之上，提供已登入使用者管理商品訂單生命週期、追蹤進貨庫存、以及處理出貨流程的功能。核心業務流程為：先有客戶訂單，再依據訂單需求進行採購（進貨），最後安排出貨。

## 術語表

- **System**：電子商務訂單管理系統整體
- **Order_Manager**：負責訂單 CRUD、狀態管理、進貨採購及出貨管理的模組
- **Product_Registry**：負責商品基本資料管理的模組
- **Customer_Registry**：負責客戶基本資料管理的模組
- **Supplier_Registry**：負責供應商（進貨商）基本資料管理的模組
- **Order**：客戶下單的銷售訂單，包含訂單編號、客戶資訊、商品明細（含明細狀態、實際供應商、實際採購成本、訂購/收到/出貨日期時間）、金額、訂單狀態
- **Product**：商品基本資料，包含名稱、SKU、單價、預設進貨成本、預設供應商、庫存數量、商品照片（多張，存放於 S3）、規格選項（如顏色、尺寸）
- **Product_Variant**：商品規格組合（如「黑 L」、「白 XL」），每個規格組合有獨立的 SKU、庫存數量、單價（可選覆寫）
- **Customer**：客戶基本資料，包含名稱、聯絡人、電話、Email、地址
- **Supplier**：供應商（進貨商）基本資料，包含名稱、聯絡人、電話、Email、地址
- **Authenticated_User**：已通過身份驗證的系統使用者

## 需求

### 需求 1：客戶管理

**使用者故事：** 身為管理人員，我希望能管理客戶基本資料，以便在建立訂單時選取正確的客戶並追蹤客戶相關交易。

#### 驗收條件

1. Customer_Registry 應提供分頁列表檢視，顯示所有客戶的名稱、聯絡人、電話、Email 及地址。
2. 當已驗證使用者提交有效的新客戶表單時，Customer_Registry 應建立該客戶並指派唯一識別碼。
3. 當已驗證使用者提交更新的客戶表單時，Customer_Registry 應更新對應的客戶記錄。
4. 若客戶建立或更新請求缺少必填欄位，Customer_Registry 應顯示驗證錯誤訊息，指出缺少的欄位。
5. Customer_Registry 應允許已驗證使用者依名稱、聯絡人或電話搜尋客戶。
6. 建立訂單時，Order_Manager 應從 Customer_Registry 中選取既有客戶，而非手動輸入客戶資訊。

### 需求 2：供應商管理

**使用者故事：** 身為管理人員，我希望能管理供應商（進貨商）基本資料，以便在建立進貨記錄時選取正確的供應商並追蹤採購來源。

#### 驗收條件

1. Supplier_Registry 應提供分頁列表檢視，顯示所有供應商的名稱、聯絡人、電話、Email 及地址。
2. 當已驗證使用者提交有效的新供應商表單時，Supplier_Registry 應建立該供應商並指派唯一識別碼。
3. 當已驗證使用者提交更新的供應商表單時，Supplier_Registry 應更新對應的供應商記錄。
4. 若供應商建立或更新請求缺少必填欄位，Supplier_Registry 應顯示驗證錯誤訊息，指出缺少的欄位。
5. Supplier_Registry 應允許已驗證使用者依名稱、聯絡人或電話搜尋供應商。
6. 對訂單明細項目執行進貨操作時，Order_Manager 應從 Supplier_Registry 中選取既有供應商，而非手動輸入供應商資訊。

### 需求 3：商品管理

**使用者故事：** 身為管理人員，我希望能管理商品基本資料，以便在訂單、進貨、出貨中引用正確的商品資訊。

#### 驗收條件

1. Product_Registry 應提供分頁列表檢視，顯示所有商品的名稱、SKU、單價、預設進貨成本、預設供應商及目前庫存數量。
2. 當已驗證使用者提交有效的新商品表單時，Product_Registry 應建立該商品並指派唯一識別碼。
3. 當已驗證使用者提交更新的商品表單時，Product_Registry 應更新對應的商品記錄。
4. 若商品建立或更新請求缺少必填欄位，Product_Registry 應顯示驗證錯誤訊息，指出缺少的欄位。
5. Product_Registry 應允許已驗證使用者依名稱或 SKU 搜尋商品。
6. 商品的預設供應商應從 Supplier_Registry 中選取既有供應商。
7. 對訂單明細項目執行進貨操作時，Order_Manager 應以商品的預設供應商作為供應商的預設值，使用者可覆寫。
8. 對訂單明細項目執行進貨操作時，Order_Manager 應以商品的預設進貨成本作為單位成本的預設值，使用者可覆寫。
9. 當已驗證使用者在商品表單中上傳一或多張照片時，Product_Registry 應將照片儲存至 S3，並將照片 URL 記錄於商品資料中。
10. Product_Registry 應支援上傳多張商品照片，並在商品詳情頁面顯示所有已上傳的照片。
11. 已驗證使用者應可刪除商品的個別照片，Product_Registry 應同時移除 S3 中的檔案與商品資料中的照片記錄。
12. 上傳照片前，系統應在前端壓縮圖片至合理尺寸，減少上傳時間與儲存成本。
13. 上傳完成後，系統應自動產生縮圖供列表頁面與預覽使用，詳情頁面點擊後載入原圖。
14. 上傳照片時，系統應為 S3 物件加上 productId 標籤，方便未來批次清理或資料分析。
15. Product_Registry 應允許已驗證使用者為商品定義多組規格維度（例如「顏色」與「尺寸」），每組維度包含多個選項值（例如顏色：紅、黑、白；尺寸：XL、L、M）。
16. Product_Registry 應根據規格維度的選項值自動產生所有規格組合（Product_Variant），例如「紅 XL」、「紅 L」、「黑 XL」等。
17. 每個 Product_Variant 應具備獨立的 SKU（可自訂或系統自動產生）及獨立的庫存數量。
18. 每個 Product_Variant 可選擇性覆寫商品的預設單價與預設進貨成本，若未覆寫則沿用商品的預設值。
19. Product_Registry 應在商品詳情頁面以表格形式顯示所有規格組合及其 SKU、單價、進貨成本、庫存數量。

### 需求 4：訂單建立與管理

**使用者故事：** 身為管理人員，我希望能建立與管理客戶銷售訂單，以便追蹤每筆交易的狀態與明細。

#### 驗收條件

1. 當已驗證使用者提交有效的新訂單表單時，Order_Manager 應建立狀態為「pending」的訂單並指派唯一訂單編號。
2. Order_Manager 應提供分頁列表檢視，顯示所有訂單的訂單編號、客戶名稱、總金額、狀態及建立日期。
3. 當已驗證使用者選取一筆訂單時，Order_Manager 應顯示訂單詳情，包含客戶資訊、明細項目（商品名稱、規格組合、數量、單價、小計、明細狀態、實際供應商、實際採購成本、相關日期）。
4. 每筆訂單明細項目應具備獨立的明細狀態，支援以下值：「待處理」、「已訂購」、「已收到」、「已出貨」、「缺貨」。
5. 新建立的明細項目初始狀態應為「待處理」。
6. 當已驗證使用者對明細項目執行進貨採購操作後，該明細項目狀態應變更為「已訂購」，並記錄訂購日期時間。
7. 當明細項目的採購記錄確認入庫後，該明細項目狀態應變更為「已收到」，並記錄收到日期時間。
8. 當已驗證使用者對明細項目執行出貨操作後，該明細項目狀態應變更為「已出貨」，並記錄出貨日期時間。
9. 當已驗證使用者將明細項目標記為缺貨時，該明細項目狀態應變更為「缺貨」。
10. 明細狀態的允許轉換順序為：待處理 → 已訂購 → 已收到 → 已出貨，或待處理 → 缺貨，或已訂購 → 缺貨。
11. 當已驗證使用者新增明細項目至訂單時，Order_Manager 應計算該明細項目小計（數量乘以單價），並更新訂單總金額。
12. 當已驗證使用者新增明細項目時，若選取的商品具有規格組合，Order_Manager 應要求使用者選取特定的 Product_Variant（例如「黑 L」或「白 XL」），明細項目的單價應使用該規格組合的單價。
13. 若選取的商品具有規格組合但使用者未選取特定規格，Order_Manager 應顯示錯誤訊息，要求選取規格。
14. 若訂單建立請求缺少必填欄位，Order_Manager 應顯示驗證錯誤訊息，指出缺少的欄位。
15. Order_Manager 應允許已驗證使用者依訂單編號或客戶名稱搜尋訂單。

### 需求 5：訂單狀態管理

**使用者故事：** 身為管理人員，我希望能更新訂單狀態，以便追蹤訂單從建立到完成的完整流程。

#### 驗收條件

1. Order_Manager 應支援以下訂單狀態：「pending」（待處理）、「confirmed」（已確認）、「shipping」（出貨中）、「completed」（已完成）、「cancelled」（已取消）。
2. 當已驗證使用者變更訂單狀態時，Order_Manager 應驗證該轉換遵循允許的順序：pending → confirmed → shipping → completed，或任何狀態 → cancelled。
3. 若已驗證使用者嘗試無效的狀態轉換，Order_Manager 應顯示錯誤訊息，說明該轉換不被允許。
4. 當訂單狀態變更時，Order_Manager 應記錄先前狀態、新狀態及變更時間戳記。
5. 當訂單中任一明細項目狀態變更為「已出貨」時，Order_Manager 應自動將訂單狀態更新為「shipping」（若尚未為 shipping 或 completed）。
6. 當訂單所有明細項目狀態皆為「已出貨」時，Order_Manager 應自動將訂單狀態更新為「completed」。

### 需求 6：訂單明細進貨採購與入庫

**使用者故事：** 身為管理人員，我希望能直接在訂單明細項目上執行進貨採購操作，指定供應商與數量後系統記錄採購資訊，入庫確認後自動增加庫存。

#### 驗收條件

1. 當已驗證使用者對訂單明細項目執行進貨操作並指定供應商、採購數量及單位成本時，Order_Manager 應建立該明細項目的一筆採購記錄，狀態為「pending」（待入庫）。
2. 同一訂單明細項目可執行多次進貨操作（分批採購），Order_Manager 應追蹤該明細項目的累計已採購數量。
3. 若指定的採購數量超過該明細項目的未採購餘額（訂單數量減去累計已採購數量），Order_Manager 應顯示錯誤訊息並阻止操作。
4. 訂單詳情頁面應顯示每筆明細項目的所有採購記錄，包含供應商、採購數量、單位成本、採購日期及狀態。
5. 當已驗證使用者將採購記錄狀態變更為「received」（已入庫）時，Order_Manager 應將關聯商品規格組合（或無規格商品）的庫存數量增加對應的採購數量，並記錄入庫日期。
6. 入庫確認操作（庫存增加、採購記錄狀態更新、明細狀態更新）應以原子性交易執行，確保資料一致性。
7. 入庫確認後，Order_Manager 應將該採購記錄的供應商及單位成本記錄為該明細項目的實際供應商及實際採購成本。
8. 若同一明細項目有多筆採購記錄（不同供應商或不同成本），訂單詳情應分別顯示各筆採購的實際供應商與實際採購成本。
9. 若已驗證使用者嘗試將「received」狀態的採購記錄變更為「cancelled」，Order_Manager 應顯示錯誤訊息，說明已入庫的記錄無法取消。
10. Order_Manager 應支援以下採購記錄狀態：「pending」（待入庫）、「received」（已入庫）、「cancelled」（已取消）。
11. 當採購記錄狀態變更時，Order_Manager 應記錄先前狀態、新狀態及變更時間戳記。

### 需求 7：訂單明細出貨與庫存扣減

**使用者故事：** 身為管理人員，我希望能直接在訂單明細項目上標記出貨，系統自動扣減庫存並記錄出貨日期，以簡化出貨流程。

#### 驗收條件

1. 僅當明細項目狀態為「已收到」時，已驗證使用者方可對該明細項目執行出貨操作。
2. 當已驗證使用者對明細項目執行出貨操作並指定出貨數量時，Order_Manager 應更新該明細項目的已出貨數量。
3. 若指定的出貨數量超過該明細項目的未出貨餘額（訂單數量減去累計已出貨數量），Order_Manager 應顯示錯誤訊息並阻止操作。
4. 若該明細項目關聯商品規格組合（或無規格商品）的庫存數量小於指定的出貨數量，Order_Manager 應顯示庫存不足的錯誤訊息並阻止操作。
5. 出貨操作成功後，Order_Manager 應將關聯商品規格組合（或無規格商品）的庫存數量減少對應的出貨數量。
6. 出貨操作（庫存扣減與狀態更新）應以原子性交易執行，確保不會因併發操作導致庫存數量不一致。
7. 若併發出貨操作導致庫存版本衝突，系統應回傳衝突錯誤，前端重新取得最新資料後可重試。

### 需求 8：導覽與存取控制

**使用者故事：** 身為管理人員，我希望透過導覽列快速存取各管理功能，且僅限已登入使用者使用。

#### 驗收條件

1. 系統應在應用程式導覽列中顯示客戶管理、供應商管理、商品管理、訂單管理的導覽連結。
2. 當未登入的使用者嘗試存取任何管理頁面時，系統應將其重新導向至登入頁面。
3. 系統應提供儀表板頁面，顯示待處理訂單、待入庫採購記錄、待出貨明細的摘要數量。

### 需求 9：訂單分拆與合併

**使用者故事：** 身為管理人員，我希望能將同一客戶的多筆未出貨訂單合併為一筆，或將一筆訂單分拆為多筆，以便靈活調整出貨安排。

#### 驗收條件

1. 當已驗證使用者選取同一客戶的兩筆或以上狀態為「pending」或「confirmed」的訂單並執行合併時，Order_Manager 應以原子性交易建立一筆新訂單，包含所有來源訂單的明細項目，並將來源訂單狀態變更為「cancelled」。
2. 合併後的新訂單總金額應等於所有來源訂單總金額的加總。
3. 若選取的訂單不屬於同一客戶，Order_Manager 應顯示錯誤訊息，說明僅能合併同一客戶的訂單。
4. 若選取的訂單中包含狀態為「shipping」、「completed」或「cancelled」的訂單，Order_Manager 應顯示錯誤訊息，說明僅能合併未出貨的訂單。
5. 當已驗證使用者對一筆狀態為「pending」或「confirmed」的訂單執行分拆，並指定明細項目的分配方式時，Order_Manager 應以原子性交易建立兩筆或以上新訂單，各自包含指定的明細項目，並將原訂單狀態變更為「cancelled」。
6. 分拆後所有新訂單的明細項目總和應等於原訂單的明細項目總和（數量守恆）。
7. 若訂單狀態為「shipping」、「completed」或「cancelled」，Order_Manager 應阻止分拆操作並顯示錯誤訊息，說明僅能分拆未出貨的訂單。

### 需求 10：資料模型序列化

**使用者故事：** 身為開發人員，我希望資料模型能正確地序列化與反序列化，以便前後端資料傳輸保持一致。

#### 驗收條件

1. 對於所有有效的 Order 物件（含明細項目及其採購/出貨記錄），系統應確保將 Order 序列化為 JSON 後再反序列化回來，產生等價的 Order 物件（往返屬性）。
2. 對於所有有效的 Product 物件，系統應確保將 Product 序列化為 JSON 後再反序列化回來，產生等價的 Product 物件（往返屬性）。
3. 對於所有有效的 Customer 物件，系統應確保將 Customer 序列化為 JSON 後再反序列化回來，產生等價的 Customer 物件（往返屬性）。
4. 對於所有有效的 Supplier 物件，系統應確保將 Supplier 序列化為 JSON 後再反序列化回來，產生等價的 Supplier 物件（往返屬性）。
