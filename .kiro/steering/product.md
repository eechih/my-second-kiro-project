# 產品概述

本專案是基於 AWS Amplify Gen 2 的 React SPA 電子商務訂單管理系統。所有使用者介面文字使用繁體中文，主要服務內部營運人員處理客戶、商品、供應商與訂單流程。

## 核心流程

系統以訂單履約為主軸，核心流程如下：

客戶下單 -> 依訂單明細進貨採購 -> 入庫確認增加庫存 -> 出貨扣減庫存 -> 訂單完成

任何程式修改若影響上述任一節點，都必須一併檢查訂單狀態、明細狀態與庫存一致性。

## 核心模組

- 驗證：Email / 密碼登入、註冊、登出與個人資料
- 客戶管理：客戶資料 CRUD、搜尋、供訂單建立時選取
- 供應商管理：供應商資料 CRUD、搜尋、供採購流程選取
- 商品管理：商品 CRUD、搜尋、多張照片上傳、規格組合與預購狀態
- 訂單管理：訂單 CRUD、明細狀態追蹤、採購、入庫、出貨、合併、分拆
- 儀表板：待處理訂單、待入庫採購與待出貨明細摘要

## 核心業務概念

### 商品規格組合

- 商品可定義多個規格維度，系統會產生對應的規格組合
- 若商品有規格組合，庫存與部分價格 / 成本邏輯以規格組合為主
- 若商品沒有規格組合，則以商品本身管理庫存

### 訂單明細狀態

- 明細狀態描述單一品項的履約進度，例如待處理、已訂購、已收到、已出貨、缺貨
- 明細狀態變更會驅動訂單主狀態更新
- 不可跳過共享邏輯直接任意覆寫明細狀態

### 訂單主狀態

- 訂單主狀態由各明細狀態推導，不應由 UI 任意指定最終值
- 任一明細出貨後，訂單可能進入出貨中；全部完成後才可完成
- 合併、分拆、取消等流程也會影響訂單主狀態

### 庫存追蹤

- 入庫會增加庫存，出貨會扣減庫存，取消流程需正確回補
- 庫存變動與明細狀態更新必須維持一致，不可只改其中一側

### 訂單合併與分拆

- 同一客戶的未出貨訂單可以合併或分拆
- 數量必須守恆，不可平白新增或遺失明細數量

## 驗證與存取控制

- `src/auth/AuthProvider.tsx` 是唯一的驗證狀態來源
- 受保護頁面透過 TanStack Router 的 `beforeLoad` 與 route context 做驗證檢查
- 未驗證使用者會被重新導向到首頁

## 產品層慣例

- 使用者面向的文字、提示與錯誤訊息一律為繁體中文
- 實體間關聯應透過選取既有記錄建立，不依賴自由輸入的文字關聯
- 複雜流程的真正規則以共享邏輯與後端交易流程為準，UI 只負責引導與呈現
